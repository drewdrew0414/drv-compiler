## dri의 보안 모델

### 컴파일러 자체 보안

dri 컴파일러는 신뢰할 수 있는 소스 파일을 입력으로 받습니다.  
악의적인 `.dri` 파일을 컴파일할 때 발생하는 문제는 보안 취약점으로 간주합니다.

특히 주의하는 영역:

- **파서 안전성**: 악의적인 입력으로 컴파일러가 충돌하거나 무한 루프에 빠지지 않아야 합니다.
- **코드 생성 안전성**: 생성된 C++ 코드가 의도치 않은 동작을 유발하지 않아야 합니다.
- **`import` 경로 탐색**: `import` 문이 허용되지 않은 경로의 파일을 포함하지 않아야 합니다.
- **`sys.exec` 인젝션**: 사용자 입력이 `sys.exec()`에 직접 전달되는 패턴에 주의합니다.

### 생성된 코드의 보안

dri가 생성한 C++ 코드는 다음을 보장합니다.

- `Own<T>` / `Ref<T>` / `Borrow<T>` 소유권 모델로 메모리 안전성 확보
- `@region` 아레나 할당으로 메모리 누수 방지
- 배열 범위 검사 (`--debug` 빌드 시 활성화)

### 알려진 제한사항

- `sys.exec(cmd)` — 사용자 입력을 그대로 전달하면 셸 인젝션 위험이 있습니다. 항상 입력을 검증하세요.
- `io.read_file(path)` — 경로 검증 없이 사용자 입력을 전달하면 경로 탐색 위험이 있습니다.
- `--release` 빌드는 배열 범위 검사를 비활성화합니다. 신뢰할 수 없는 입력을 처리할 때는 `--debug` 빌드로 먼저 검증하세요.

## 보안 관련 어노테이션

dri는 보안·안전성을 명시적으로 표현하는 어노테이션을 제공합니다.

```dri
@threadsafe
void safe_increment() {
    counter += 1;
};

@io
String read_config(String path) {
    return io.read_file(path);
};

@alloc
double[] create_buffer(int n) {
    return new double[n];
};
```

이러한 이펙트 어노테이션은 코드 리뷰 시 보안 취약점을 더 쉽게 식별하는 데 도움을 줍니다.
