# 메모리 모델
> drv는 Rust에서 영감을 받은 소유권·대여 모델을 제공한다. `Own`, `Ref`, `Borrow`, `@region`으로 메모리를 제어한다.

---

## 소유권 타입 요약

| 타입 | C++ 대응 | 소유권 | 해제 | 동시 접근 |
|------|----------|--------|------|----------|
| `Own<T>` | `unique_ptr<T>` | 단독 (move 가능) | 자동 | 단일 스레드 |
| `Ref<T>` | `shared_ptr<T>` | 공유 | 카운트 0시 자동 | 불변 공유 가능 |
| `Borrow<T>` | `const T&` | 없음 | 해제 안 함 | 동시 읽기 가능 |
| `mut Borrow<T>` | `T&` | 없음 | 해제 안 함 | 독점 쓰기 |
| `ref T` | `T&` | 없음 | 해제 안 함 | 별칭 |
| `@region` | arena | 영역 내 전체 | 블록 종료 시 일괄 | 탈출 금지 |

> 기본 타입의 `Ref<T>`, `Own<T>`, `Borrow<T>` 선언 문법은 [[02_기본.md](02_기본.md)]를 참고하세요.

## Ref\<T\> — 공유 소유권 (shared_ptr)

여러 변수가 같은 객체를 공유합니다. null을 허용합니다.

```drv
Ref<String> a = new String();
Ref<MyClass> obj = new MyClass();

# null 허용
Ref<String> maybe = null;
if (maybe != null) {
    print(maybe);
};
```

연결 리스트 예시:

```drv
class Node {
    public int val;
    public Ref<Node> next;
};

Ref<Node> head = new Node();
head.val = 1;
head.next = new Node();
head.next.val = 2;
head.next.next = null;

Ref<Node> cur = head;
while (cur != null) {
    print(cur.val);
    cur = cur.next;
};
```

## Own\<T\> — 단일 소유권 (unique_ptr)

한 번에 하나의 변수만 소유합니다. 소유권 이전은 반드시 **`move` 키워드를 명시**해야 합니다.  
`move` 없이 `Own<T>`를 대입하면 **컴파일 오류**입니다 — 암묵적 이전은 허용하지 않습니다.

```drv
Own<int[]> buf = new int[1024];

# 명시적 move — 소유권 이전
Own<int[]> owner2 = move buf;
# 이후 buf는 무효화(invalidated) — 접근 불가

print(buf[0]);   # 컴파일 오류: 'buf' used after move
```

### move 규칙

| 상황 | 동작 |
|------|------|
| `move` 명시 | 소유권 이전, 원본 무효화 |
| `move` 없이 `Own<T>` 대입 | **컴파일 오류** |
| 무효화된 변수 접근 | **컴파일 오류** |
| `Ref<T>` / `Borrow<T>` 대입 | `move` 불필요 (공유/대여) |

```drv
# 오류 예시
Own<int[]> a = new int[64];
Own<int[]> b = a;         # 오류: move 키워드 없이 Own<T> 대입 불가
Own<int[]> c = move a;    # OK
print(a[0]);              # 오류: 'a' used after move
```

함수로 소유권 이전:

```drv
void consume(Own<int[]> data) {
    data[0] = 42;
    print(data[0]);
};

Own<int[]> arr = new int[512];
consume(move arr);         # arr 소유권이 consume으로 이동
consume(move arr);         # 오류: 'arr' used after move
```

함수에서 소유권 반환 (돌려받기):

```drv
Own<int[]> process(Own<int[]> data) {
    data[0] = 99;
    return move data;   # 소유권 반환
};

Own<int[]> buf = new int[128];
buf = process(move buf);   # 넘겼다가 돌려받음 — buf 다시 유효
print(buf[0]);             # 99
```

### 컴파일러 오류 메시지 규정

| 상황 | 오류 메시지 |
|------|-----------|
| `move` 없이 `Own<T>` 대입 | `error: cannot copy 'Own<T>' — use 'move' to transfer ownership` |
| 무효화 후 접근 | `error: 'X' used after move at line N` |
| `move`된 변수 재사용 | `error: value moved into 'Y', 'X' is no longer valid` |

## Borrow\<T\> — 비소유 참조 (const T&)

복사 없이 읽기 전용으로 접근합니다. 소유권이 이전되지 않습니다.

```drv
void print_buf(Borrow<int[]> buf) {
    print(buf[0], buf[1]);
};

Own<int[]> data = new int[4];
data[0] = 10;
data[1] = 20;

print_buf(data);    # 10 20
# data의 소유권은 그대로 유지
```

## mut Borrow\<T\> — 가변 독점 대여 (T&, exclusive)

읽기 전용 `Borrow<T>`와 달리, **하나의 가변 대여만** 동시에 존재할 수 있습니다.  
이 규칙으로 `parallel for` 등 병렬 컨텍스트에서 **데이터 경쟁(Data Race)을 컴파일 타임에 방지**합니다.

### 대여 규칙 (Read-Write Lock 철학)

| 상태 | 허용 |
|------|------|
| 불변 대여 `Borrow<T>` 여러 개 동시 존재 | ✅ |
| 가변 대여 `mut Borrow<T>` 하나만 존재 | ✅ |
| 가변 대여 중 다른 불변/가변 대여 동시 존재 | ❌ 컴파일 오류 |

```drv
Own<int[]> data = new int[1024];

void read_only(Borrow<int[]> b) { print(b[0]); };
void write(mut Borrow<int[]> b) { b[0] = 99; };

read_only(data);    # OK
read_only(data);    # OK — 불변 대여 여러 개 허용

write(data);        # OK
write(data);        # OK — 순차 실행이므로 동시 가변 대여 아님

# 컴파일 오류 예시 — 가변 대여 중 불변 대여 불가
mut Borrow<int[]> w = data;
Borrow<int[]> r = data;   # 오류: w가 살아있는 동안 r 생성 불가
```

### `parallel for` 안전성

```drv
int[] arr = [1, 2, 3, 4, 5];

# 안전: 각 스레드가 독립 원소를 불변으로 읽음
parallel for (item of arr) {
    print(item * 2);
};

# 컴파일 오류: 루프 내부에서 arr 자체를 수정하려 하면 가변 대여 충돌
parallel for (item of arr) {
    arr[0] = 0;   # 오류: arr이 병렬로 가변 대여됨
};
```

## ref — 참조 변수 (T&)

```drv
int x = 10;

print(x);        # 20
```

## @region — 아레나 할당

```
@region 이름 {
    본문
}
```

블록 내에서 할당한 모든 메모리를 블록 종료 시 일괄 해제합니다.

```drv
@region scratch {
    int[] temp = [0, 0, 0, 0, 0, 0, 0, 0];
    double[] workspace = [1.0, 2.0, 3.0, 4.0];

    for (i in 0..8) {
        temp[i] = i * 2;
    };

    int s = 0;
    for (v of temp) {
        s += v;
    };
    print("합:", s);
    # 블록 종료: temp, workspace 일괄 해제
}
```

### 중첩 region

```drv
@region outer {
    int[] big_buf = new int[10000];

    @region inner {
        int[] small_tmp = new int[64];
        # 연산 수행
        # inner 종료: small_tmp 즉시 해제
    }
    # big_buf는 아직 유효
    # outer 종료: big_buf 해제
}
```

### HPC 패턴 — 반복 임시 할당

```drv
for (iter in 0..1000) {
    @region frame {
        double[] grad = new double[1024];
        double[] update = new double[1024];

        simd for (i in 0..1024) {
            grad[i] = compute_grad(i, iter);
            update[i] = grad[i] * 0.01;
        };
        # frame 종료: 매 반복마다 힙 단편화 없이 일괄 해제
    }
};
```

### @region 탈출 분석 (Escape Analysis)

`@region` 블록 내부에서 할당된 메모리 또는 `io.mmap_read`로 얻은 파일 매핑이  
**블록 외부의 변수에 대입(탈출)되면 컴파일 오류**입니다.  
블록 종료 시 메모리가 해제되므로, 외부 참조는 Use-After-Free 취약점을 유발합니다.

```drv
# 오류 예시 — mmap 결과를 region 밖으로 탈출
Ref<var> escaped;

@region frame {
    var mapped = io.mmap_read("huge.bin");
    escaped = mapped;   # 컴파일 에러: mapped가 @region frame 밖으로 탈출
}
# 이 시점에 frame 종료 → mapped 해제됨
# escaped는 해제된 메모리를 가리키는 댕글링 포인터
print(escaped);   # Use-After-Free
```

```drv
# 올바른 패턴 — 블록 안에서만 사용
@region frame {
    var mapped = io.mmap_read("huge.bin");
    process(mapped);          # 블록 안에서만 참조
    double result = sum(mapped);
    print("에너지:", result);
    # frame 종료: mapped 안전하게 해제
}
```

> 탈출 분석은 컴파일러가 자동 수행합니다. 탈출 감지 시 오류 메시지:  
> `error: 'mapped' escapes @region 'frame' — use-after-free would occur`

## @align — SIMD/NUMA 메모리 정렬

SIMD 명령어(`simd for`, `simd.fmadd`)는 데이터가 특정 바이트 경계에 정렬되어 있을 때 최고 성능을 냅니다.  
정렬되지 않은 메모리(Misaligned Access)는 성능 저하 또는 크래시를 유발합니다.

| SIMD 명령어 | 필요 정렬 |
|------------|----------|
| SSE2 | 16바이트 |
| AVX2 | 32바이트 |
| AVX-512 | 64바이트 |

```drv
# AVX-512 최적화를 위해 64바이트 정렬 강제
@align(64)
double[] simd_buf = new double[1000000];

# AVX2용 32바이트 정렬 구조체
@align(32)
class Vector3D {
    public double x;
    public double y;
    public double z;
};
```

> C++ 트랜스파일 시 `alignas(N)` 또는 `__attribute__((aligned(N)))`으로 변환됩니다.

## compile_eval — 컴파일 타임 실행

단순 상수 계산:

```drv
int BUF_SIZE = compile_eval(1024 * 1024);
double PI_4 = compile_eval(3.14159 / 4.0);
print(BUF_SIZE);   # 1048576
```

### compile_eval 함수 블록 — 룩업 테이블 생성

`compile_eval` 어노테이션을 함수에 붙이면 컴파일 타임과 런타임 모두 실행 가능한 함수가 됩니다.  
`const` 변수에 대입 시 **컴파일 타임에 실행**되어 결과가 바이너리 데이터 섹션에 상수로 박힙니다.  
C++17 `constexpr`을 완벽히 래핑합니다.

```drv
compile_eval double[] generate_sin_table() {
    mut double[] table = new double[360];
    for (i in 0..360) {
        table[i] = math.sin(i as double * 0.017453);   # 도 → 라디안
    };
    return table;
};

# 컴파일 타임에 테이블 생성 → 바이너리에 상수로 박힘 (런타임 오버헤드 0)
const double[] sin_table = generate_sin_table();

# 런타임에 즉시 조회
print(sin_table[90]);   # ~1.0
```

```drv
compile_eval int[] generate_primes(int limit) {
    mut int[] primes = [];
    for (n in 2..limit) {
        boolean is_prime = true;
        for (p of primes) {
            if (p * p > n) { break; };
            if (n % p == 0) { is_prime = false; break; };
        };
        if (is_prime) { lst.push(primes, n); };
    };
    return primes;
};

const int[] small_primes = generate_primes(1000);   # 컴파일 타임 계산
```

## 이펙트 어노테이션

함수가 어떤 부작용을 가지는지 명시합니다.

```drv
@alloc
int[] make_buf(int n) {
    return new int[n];
};

@io
void read_line() {
    String s;
    input(s);
};

@threadsafe
void increment() {
    counter += 1;
};
```
