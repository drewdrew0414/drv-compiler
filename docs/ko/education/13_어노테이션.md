# 어노테이션
> 어노테이션은 `@` 접두사로 컴파일러에게 최적화·레이아웃·이펙트 힌트를 제공한다.

---

## 전체 목록

| 어노테이션 | 적용 대상 | 설명 |
|------------|----------|------|
| `@inline` | 함수 | 인라인 강제 |
| `@pure` | 함수 | 순수 함수 (부작용 없음) |
| `@fastcall` | 함수 | 빠른 호출 규약 |
| `@abstract` | 메서드 | 자식 클래스 구현 강제 |
| `@stack` | 변수 | 스택 강제 할당 |
| `@heap` | 변수 | 힙 명시 할당 |
| `@local` | 변수 | 스택/아레나 강제 |
| `@align(N)` | 변수·클래스 | N바이트 정렬 강제 (SIMD/NUMA 최적화) |
| `@packed` | 클래스 | 패딩 없는 밀착 레이아웃 (상속 불가) |
| `@layout_soa` | 배열 변수 | Structure of Arrays 레이아웃 강제 |
| `@noalias` | 함수 | 포인터 앨리어싱 없음 (`restrict`) |
| `@io` | 함수 | I/O 이펙트 명시 |
| `@alloc` | 함수 | 메모리 할당 이펙트 명시 |
| `@threadsafe` | 함수 | 스레드 안전 명시 |
| `@bench` | 함수 | 실행 시간 자동 측정 |
| `@specialize` | 함수 | 타입별 SIMD 특수화 |
| `@trace` | 함수 | 분기 추적 인스트루멘테이션 자동 주입 |
| `@warrant` | 함수·규칙 | 툴민 논증의 Warrant 관계 명시 |
| `@rebuttal` | 함수·규칙 | 툴민 논증의 Rebuttal 관계 명시 |
| `@defeats` | 함수·규칙 | 툴민 논증의 Defeat 관계 명시 |

## 함수 최적화

### @inline — 인라인 강제

함수 호출 오버헤드를 제거합니다.

```dri
@inline
int square(int x) {
    return x * x;
};

@inline
double clamp(double val, double lo, double hi) {
    if (val < lo) { return lo; };
    if (val > hi) { return hi; };
    return val;
};
```

### @pure — 순수 함수

부작용이 없고 같은 입력에 항상 같은 출력을 반환합니다. 컴파일러가 결과를 캐시하거나 호출을 제거할 수 있습니다.

```dri
@pure
double normalize(double x, double min, double max) {
    return (x - min) / (max - min);
};

@pure
int gcd(int a, int b) {
    while (b != 0) {
        int t = b;
        b = a % b;
        a = t;
    };
    return a;
};
```

### @fastcall — 빠른 호출 규약

핫 경로 함수에 적용합니다.

```dri
@fastcall
void process(int x, int y, int z) {
    print(x + y + z);
};

@fastcall
double lerp(double a, double b, double t) {
    return a + (b - a) * t;
};
```

## 메모리 레이아웃

### @stack — 스택 할당 강제

```dri
void compute() {
    @stack
    double[] local_buf = [0.0, 0.0, 0.0, 0.0];

    simd for (i in 0..4) {
        local_buf[i] = i as double * 2.0;
    };
};
```

### @heap — 힙 명시 할당

```dri
@heap
double[] large = new double[10000000];
```

### @packed — 패딩 없는 구조체

필드 사이 패딩을 제거해 메모리를 절약합니다.

```dri
@packed
class Pixel {
    public char r;
    public char g;
    public char b;
    public char a;    # 정확히 4바이트
};

@packed
class Vertex {
    public float x;
    public float y;
    public float z;   # 정확히 12바이트
};
```

> **제약**: `@packed`는 상속이 없는 **리프 데이터 클래스**에만 사용할 수 있습니다.  
> 상속 클래스(`extends`)에 적용하거나, `@packed` 클래스를 상속하면 **컴파일 오류**입니다.  
> 이유: 상속 시 삽입되는 vtable 포인터의 정렬이 `@packed`로 인해 깨져  
> Misaligned Access(성능 저하 또는 크래시)가 발생합니다.

### @align(N) — SIMD/NUMA 메모리 정렬

SIMD 벡터 명령어는 데이터가 특정 바이트 경계에 정렬되어 있을 때 최고 성능을 냅니다.

```dri
# AVX-512 최적화 (64바이트 정렬)
@align(64)
double[] simd_buf = new double[1000000];

# AVX2 최적화 (32바이트 정렬)
@align(32)
class Vector3D {
    public double x;
    public double y;
    public double z;
};
```

C++ 변환: `alignas(N)` 또는 `__attribute__((aligned(N)))`

| 정렬 | 대상 SIMD |
|------|----------|
| `@align(16)` | SSE2 |
| `@align(32)` | AVX2 |
| `@align(64)` | AVX-512 |

### @layout_soa — Structure of Arrays 레이아웃

SIMD 루프나 GPU 업로드에 최적인 SoA 레이아웃으로 배열을 저장합니다.  
소스 코드는 AoS처럼 작성하지만 컴파일러가 내부 레이아웃만 변환합니다.

```dri
class Particle {
    public double x;
    public double y;
    public double z;
};

# 내부: double[] x, double[] y, double[] z 배열 3개로 분리 저장
@layout_soa
Particle[] pts = new Particle[1000];

# 소스 코드는 동일, 내부적으로 x[i] += ... 형태로 SIMD 최적화됨
simd for (i in 0..1000) {
    pts[i].x += pts[i].mass * 0.01;
};
```

### @local — 스택/아레나 강제

```dri
@local
int[] buf = new int[64];
```

## 벡터화

### @noalias — 포인터 앨리어싱 없음

서로 다른 포인터가 같은 메모리를 가리키지 않음을 선언합니다. `simd for`와 함께 사용하면 더 공격적인 벡터화가 가능합니다.

```dri
@noalias
void vec_add(double[] a, double[] b, double[] c, int n) {
    simd for (i in 0..n) {
        c[i] = a[i] + b[i];
    };
};

@noalias
void scale(double[] v, double factor, int n) {
    simd for (i in 0..n) {
        v[i] = v[i] * factor;
    };
};
```

## 이펙트 시스템

함수가 어떤 부작용을 가지는지 명시합니다.

### @io — I/O 수행

```dri
@io
String read_config(String path) {
    return io.read_file(path);
};

@io
void log(String msg) {
    io.append_file("app.log", msg + "\n");
    print(msg);
};
```

### @alloc — 메모리 할당

```dri
@alloc
double[] create_buf(int n) {
    return new double[n];
};

@alloc
Map<String, int> build_index(list<String> keys) {
    Map<String, int> idx = {};
    for (i in 0..lst.size(keys)) {
        map.set(idx, lst.get(keys, i), i);
    };
    return idx;
};
```

### @threadsafe — 스레드 안전

```dri
int counter = 0;

@threadsafe
void safe_increment() {
    counter += 1;
};

@threadsafe
int safe_get() {
    return counter;
};
```

## 추상 메서드

### @abstract — 구현 강제

추상 클래스 내에서 자식 클래스가 반드시 구현해야 할 메서드를 표시합니다.

```dri
abstract class Codec {
    @abstract String encode(String input);
    @abstract String decode(String encoded);

    void verify(String original) {
        String enc = this.encode(original);
        String dec = this.decode(enc);
        if (dec == original) {
            print("검증 통과");
        } else {
            print("검증 실패");
        };
    };
};

class Base64Codec extends Codec {
    override String encode(String input) {
        return input;
    };

    override String decode(String encoded) {
        return encoded;
    };
};
```

## 어노테이션 조합

여러 어노테이션을 함께 사용할 수 있습니다.

```dri
@inline
@pure
@fastcall
double fast_lerp(double a, double b, double t) {
    return a + (b - a) * t;
};

@noalias
@io
void stream_write(double[] data, int n, String path) {
    for (i in 0..n) {
        io.append_file(path, str.from_int(data[i] as int) + "\n");
    };
};

@alloc
@pure
int[] make_range(int n) {
    int[] arr = new int[n];
    for (i in 0..n) {
        arr[i] = i;
    };
    return arr;
};
```

## 분기 추적 — `@trace`

함수 내 모든 `if/else`, `switch/match` 진입점에 **인스트루멘테이션 코드를 자동 주입**합니다.  
각 분기 진입 시 `(함수명, 조건 문자열, 라인 번호)`를 스레드 로컬 맵에 기록합니다.  
`sys.get_branch_trace()`로 현재 스레드의 분기 이력을 조회할 수 있습니다.

```dri
@trace
Result<double> safe_divide(double a, double b) {
    if (b == 0.0) {
        return Err("분모가 0");
    };
    return Ok(a / b);
};

safe_divide(10.0, 0.0);

# 분기 이력 조회
list<String> trace = sys.get_branch_trace();
for (entry of trace) {
    print(entry);
};
# 출력: "safe_divide | b == 0.0 → true | line 3"
```

C++ 변환 시 각 분기 앞에 `__dri_trace_push(__func__, "b == 0.0", __LINE__)` 형태로 주입됩니다.  
`dri-toulmin` 라이브러리는 이 이력을 논증 근거 트리 구성에 활용합니다.

## 툴민 논증 어노테이션 — `@warrant`, `@rebuttal`, `@defeats`

`dri-toulmin` 라이브러리와 연동하여 함수·규칙 간의 **논증 관계 그래프**를 바이너리에 메타데이터로 포함시킵니다.  
런타임 룰 엔진이 리플렉션으로 이 메타데이터를 읽어 신뢰도 점수를 계산합니다.

> 참조: [park-jun-woo/toulmin](https://github.com/park-jun-woo/toulmin)

| 어노테이션 | 툴민 요소 | 의미 |
|-----------|----------|------|
| `@warrant` | Warrant | "이 함수가 참이면 주장을 지지한다" |
| `@rebuttal` | Rebuttal | "이 함수가 참이면 주장에 예외를 제기한다" |
| `@defeats` | Defeat | "이 함수가 참이면 다른 규칙을 무력화한다" |

```dri
# 기본 주장 (Claim)
boolean is_safe_speed(double speed_kmh) {
    return speed_kmh <= 100.0;
};

# Warrant: 과속이면 위험하다는 주장을 지지
@warrant("is_safe_speed")
boolean no_speeding(double speed_kmh) {
    return speed_kmh < 120.0;
};

# Rebuttal: 고속도로에서는 예외
@rebuttal("is_safe_speed")
boolean highway_exception(String road_type) {
    return road_type == "highway";
};

# Defeats: 긴급차량이면 no_speeding 규칙 무력화
@defeats("no_speeding")
boolean emergency_vehicle(boolean is_emergency) {
    return is_emergency;
};
```

컴파일러는 이 어노테이션을 파싱해 바이너리에 다음 메타데이터를 포함합니다:

```json
{
  "rules": [
    { "name": "no_speeding",       "type": "warrant",  "target": "is_safe_speed" },
    { "name": "highway_exception", "type": "rebuttal", "target": "is_safe_speed" },
    { "name": "emergency_vehicle", "type": "defeats",  "target": "no_speeding" }
  ]
}
```

> `dri-toulmin` 라이브러리 연동 및 신뢰도 계산 알고리즘은 [[17_외부라이브러리.md](17_외부라이브러리.md)]를 참고하세요.
