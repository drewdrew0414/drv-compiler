## 脆弱性の報告

セキュリティ脆弱性を発見した場合は、**公開 Issue ではなく非公開で報告**してください。

### 報告方法

**メール**: `drew0414drew@gmail.com`  
件名形式: `[SECURITY] dri - <脆弱性の概要>`

### 含めるべき情報

1. 脆弱性の種類 (例: バッファオーバーフロー、パストラバーサル、コードインジェクション)
2. 影響を受けるコンポーネント (Lexer, Parser, Codegen など)
3. 最小再現例 (`.dri` コードまたは CLI コマンド)
4. 想定される影響範囲

## dri のセキュリティモデル

### コンパイラ自体のセキュリティ

dri コンパイラは、信頼できるソースファイルを入力として受け取ることを前提としています。  
悪意ある `.dri` ファイルがコンパイラをクラッシュ・ハング・乗っ取らせる場合、それは脆弱性として扱います。

特に注意している領域:

- **パーサの堅牢性**: 悪意ある入力で無限ループ・クラッシュしないこと。
- **コード生成の安全性**: 生成された C++ が意図しない動作を引き起こさないこと。
- **`use` のパス解決**: モジュールインポートが許可されていないパスのファイルを読み込まないこと。
- **`sys.exec` インジェクション**: ユーザー入力が `sys.exec()` に直接渡るパターンへの警戒。

### 生成コードのセキュリティ

dri が生成する C++ は以下を保証します:

- `Own<T>` / `Ref<T>` / `Borrow<T>` 所有権モデルによるメモリ安全性。
- `@region` アリーナアロケーションによるリーク防止。
- 配列範囲チェック (`--debug` ビルド時に有効)。

### 既知の制約

- `sys.exec(cmd)` — ユーザー入力を直接渡すとシェルインジェクションのリスクがあります。必ず入力を検証してください。
- `io.read_file(path)` — 検証なしのユーザー入力はパストラバーサルのリスクがあります。
- `--release` ビルドは範囲チェックを無効化します。信頼できない入力を扱う際は、まず `--debug` で検証してください。

## セキュリティ関連のアノテーション

dri はセキュリティ・安全性を明示するアノテーションを提供します。

```dri
@threadsafe
void safe_increment() {
    counter += 1;
};

@io
String read_config(String path) {
    return io.read_file(path);
};

@alloc
double[] create_buffer(int n) {
    return new double[n];
};
```

これらの効果アノテーションにより、コードレビューで危険な操作を発見しやすくなります。
